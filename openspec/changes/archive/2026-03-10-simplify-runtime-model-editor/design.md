## Context

当前运行时模型配置已经以 `api_sources + runtime_model_settings` 为真相源，但“模型设置”编辑器仍沿用较重的技术字段视图：用户需要同时输入模型 ID、显示名称、协议族、实际模型名，并通过 runtime 级下拉指定默认模型。用户反馈这些字段与操作不符合多 CLI 管理的心智模型，因为大多 runtime 实际只需要“这个模型是什么、显示成什么、走哪个来源”，默认模型也更适合在卡片上直接设定。

同时，后端保存结构仍要求 `provider` 与 `model` 两个字段，运行时配置解析与测试调用也依赖这两个字段。因此本次调整需要在尽量不破坏现有持久化结构的前提下，让 UI 可以隐藏这些高级字段。

## Goals / Non-Goals

**Goals:**
- 将“模型设置”页简化为三项核心输入：模型、显示名称、API 来源。
- 让每个 runtime 的模型以三列卡片网格呈现，并在卡片头部直接执行默认、测试、删除操作。
- 保持现有持久化结构和运行时解析能力不变，避免影响聊天页、Repo Library、CLI 启动链路。
- 让后端在保存时兼容简化 payload，减少 UI 与存储层的耦合。

**Non-Goals:**
- 不调整 runtime / source 的整体分层结构。
- 不引入跨 runtime 的全局模型 ID 唯一性约束。
- 不移除持久化结构中的 `provider`、`model` 字段。
- 不重构 `POST /api/v1/settings/llm/test` 的接口形态。

## Decisions

### 1. 保留持久化结构，隐藏高级编辑字段
- 决策：前端不再展示协议族与实际模型名，但后端仍持久化 `provider`、`model`。
- 原因：聊天、CLI wrapper、测试调用当前都直接依赖这两个字段；保留存储结构可以避免大范围联动修改。
- 备选方案：直接从配置模型中删除 `provider` / `model` 字段。该方案会影响运行时解析、历史配置兼容与多处调用链，风险过高。

### 2. 以后端规范化兜底推导高级字段
- 决策：`PUT /api/v1/settings/runtime-models` 保存时，如果模型缺少 `provider`，就按 `source_id` 对应来源推导；如果缺少 `model`，就使用归一化后的 `id`。
- 原因：这样前端可以只关注用户输入字段，同时接口对旧客户端和未来简化客户端都更稳健。
- 备选方案：仅在前端拼装完整 payload。该方案可行，但接口鲁棒性差，后续其他入口复用时仍需重复推导逻辑。

### 3. 将默认模型操作下沉到模型卡片
- 决策：移除 runtime 级默认模型下拉，在每张模型卡片上提供 `设为默认` 按钮，并直接更新 `default_model_id`。
- 原因：默认动作本质上属于某个模型本身，放在卡片上更易理解，也减少模型多时的跨区域跳转。
- 备选方案：保留下拉并额外提供卡片按钮。该方案会保留重复入口，继续增加界面负担。

### 4. 测试调用使用“卡片有效绑定”
- 决策：测试按钮继续走 `POST /api/v1/settings/llm/test`，但 `provider` 来自当前所选 API 来源，`model` 使用模型 ID。
- 原因：这与“模型 ID 即实际模型名”的新编辑心智保持一致，同时复用既有测试链路。
- 备选方案：新增专用 runtime-model 测试接口。该方案收益有限，当前没有必要。

## Risks / Trade-offs

- [隐藏高级字段会降低少数特殊映射场景的可见性] → 当前以“模型 ID 即模型名”为统一约定；若未来需要特例，可在后续引入高级模式，而不是保留默认复杂度。
- [跨 runtime 重复模型 ID 仍可能带来全局引用歧义] → 本次保持现有 runtime-scope 语义不变，相关解析继续以 runtime + model 绑定为准。
- [来源被切换后，推导 provider 可能与 runtime 不兼容] → 保存时继续复用现有 runtime/source 兼容校验，非法组合直接返回错误。

## Migration Plan

- 现有配置无需迁移；旧配置中的 `provider` / `model` 会继续被读取。
- 用户下一次通过新 UI 保存时，后端会将缺失的 `provider` / `model` 自动补齐为新规则。
- 如果需要回滚，只需恢复原 UI 字段和保存逻辑；后端兼容逻辑向后兼容，不影响旧数据。
