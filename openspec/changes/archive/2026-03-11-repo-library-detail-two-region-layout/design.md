## Context

当前详情页已经有报告弹窗、快照选择、分析运行选择、仓库上下文、知识卡片与卡片详情，但这些元素还在沿用“并列栏位”的阅读模型。用户现在要把页面重新组织成两个主区域：左边负责稳定上下文，右边负责“选卡 + 阅读”，并进一步压缩卡片导航里的冗余信息。

## Goals / Non-Goals

**Goals:**
- 把主工作区改成 `2:3` 的左右双区域布局。
- 让左侧区域以“参数在上、上下文在下”的方式承载全部仓库背景信息。
- 让右侧区域以“横向卡片选择带在上、详情阅读面在下”的方式承载主要交互。
- 让知识卡片选择项只保留标题与类型标签，避免和详情区重复。

**Non-Goals:**
- 不改后端接口和数据结构。
- 不新增搜索、分组、过滤等新的知识卡片交互能力。
- 不改变报告弹窗与 evidence 数据的语义。

## Decisions

### 1. 主区改成 `2:3` 双区域，而不是继续维持三栏

原因：
- 用户的主要视线路径已经收敛为“先看上下文，再看卡片与详情”，不需要三块几乎等权的并列容器。
- 左边合并参数和上下文后，右边可以形成更连续的知识阅读面。

备选：
- 保留三栏，只继续调视觉重量。
- 放弃三栏的根本问题是信息流仍然被切碎，用户已经明确不喜欢这种结构。

### 2. 左侧区域使用上下堆叠：参数盒子 + 仓库上下文

原因：
- 快照和分析运行属于仓库上下文切换，和仓库摘要天然是一组信息。
- 技术栈、后端、前端、其它模块这些内容普遍较长，需要更宽的单列阅读宽度。

### 3. 右侧顶部改成横向滚动的卡片选择带

原因：
- 卡片的摘要、结论和实现机制已经在详情区完整展示，导航层重复展示只会增加噪声。
- 横向滚动带可以同时满足“快速浏览多个卡片”和“保持右侧详情区连续纵向阅读”的目标。

方案：
- 每张卡片使用固定宽度的轻量矩形项。
- 卡片项只展示标题和类型标签。
- 使用横向滚动而不是纵向长列表，避免卡片导航把详情区向下推。

### 4. 右侧详情区保留纵向主阅读，Evidence 继续作为下半区支撑内容

原因：
- 用户仍然需要在同一个主阅读区里查看结论、实现机制和 evidence。
- 只要把卡片导航移到顶部，详情区本身不需要再承担目录职责。

## Risks / Trade-offs

- [Risk] 横向卡片带在卡片较多时可能出现定位不稳 → Mitigation：保持固定卡片宽度和明确选中态，优先保证可扫读性。
- [Risk] 左侧上下文文本较长时会挤压顶部参数盒子 → Mitigation：左列内部继续使用分区滚动，上半区内容保持紧凑。
- [Risk] 右侧顶部卡片带高度过大影响详情阅读高度 → Mitigation：卡片项只保留标题和标签，严格控制高度。
