## Context

这次改动不是单一模块重命名，而是一次跨 `backend/`、`ui/`、`desktop/`、`services/`、`scripts/`、`docs/` 和仓库目录本身的全链路迁移。当前项目同时存在三类旧标识：

- 对外品牌名：文档、UI、桌面壳、提示词和说明文字中的 `vibe-tree`
- 运行标识：可执行名、环境变量前缀、配置目录、数据目录、日志目录
- 仓库与源码标识：模块路径、脚本路径、当前工作目录名、构建产物名

这三类标识不能只改一部分，否则会出现“文案已改、运行未改”或“路径已改、导入仍旧”的不一致状态。尤其当前项目已经把 repo library、chat runtime、desktop shell 和 repo analyzer 都接入到同一个产品面，名称切换必须一次性收口。

## Goals / Non-Goals

**Goals:**

- 统一项目对外名称为 `VibeCraft`
- 统一默认运行标识为 `vibecraft`，包括可执行名、默认目录和环境变量前缀
- 迁移当前仓库目录名，并同步修正脚本、文档、Go module/import path、桌面壳和构建入口
- 为已有本地配置与数据提供兼容读取或迁移逻辑，避免升级后直接丢失状态
- 在改名后仍保持后端构建、前端构建和测试链路可运行

**Non-Goals:**

- 不在这次变更中重构产品能力边界或信息架构
- 不新增和重命名无关的 API 能力
- 不在这次变更中清理所有历史文档内容，只处理会误导当前使用的主路径与主文档

## Decisions

### 1. 对外品牌名与运行标识分层统一

决策：对外名称统一为 `VibeCraft`，运行标识统一为 `vibecraft`。

原因：品牌名和技术标识本来就应分别服务于展示与执行。继续混用 `vibe-tree` 会让 README、UI 和本地路径不一致。使用 `VibeCraft` / `vibecraft` 的双层规范，能够同时覆盖 UI 标题和 CLI/路径命名。

备选方案：

- 只改文案，不改运行标识：会留下长期技术债，用户仍会看到旧路径和旧命令
- 保留 `vibe-tree` 作为内部标识：短期省事，但后续每个新功能都要继续解释旧名

### 2. 兼容迁移优先于一次性断裂

决策：默认路径切到新前缀，但启动时优先读取或迁移旧配置/旧数据目录。

原因：这次改名会影响 `~/.config/...`、`~/.local/share/...` 等本地状态。完全断裂会直接让用户丢失已有配置和历史数据。兼容读取或启动迁移能把升级风险降到最低。

备选方案：

- 直接切新目录，不认旧目录：实现简单，但破坏性过强
- 长期双写新旧目录：兼容强，但维护复杂且容易造成状态分叉

### 3. 先修源码引用，再迁移仓库目录

决策：先完成仓库内源码与文档对新名字的适配，再在收尾阶段重命名当前目录。

原因：很多脚本和工具依赖当前工作目录。过早改目录会让中间验证过程变脆弱。先让代码和文档不再硬编码旧目录，再执行目录改名，收尾更稳。

备选方案：

- 一开始就改目录：中途命令上下文容易失效
- 永久不改目录，只改代码：与用户要求不符，也会留下明显不一致

### 4. Go module、构建名和桌面壳一起切换

决策：Go module path、二进制名、桌面壳显示名和相关脚本一并改为新标识。

原因：这些入口直接决定开发体验和发布产物名称。如果只改 README 和 UI，不改 module path 和 build name，后续仓库迁移仍然会受阻。

备选方案：

- 只保留旧 module path：短期少改，但会让 import path 永久背离项目名

## Risks / Trade-offs

- [旧数据目录迁移失败] → 启动时优先读取旧目录；必要时复制而不是移动；保留回退提示
- [Go import path 大范围改动引入编译错误] → 使用定向替换后运行 `go test ./...` 覆盖验证
- [脚本或文档仍残留旧目录名] → 用仓库级检索收尾，分“运行面”和“历史文档面”分别检查
- [目录改名影响当前会话路径] → 在大部分代码修改和验证完成后最后执行目录重命名

## Migration Plan

1. 定义新产品名、新运行前缀和迁移规则
2. 修改源码、文档、脚本、配置路径、环境变量和构建产物名
3. 增加旧路径到新路径的兼容读取或迁移逻辑
4. 运行前后端构建与测试
5. 在最后阶段将当前仓库目录重命名为新目录名
6. 复查文档与脚本中的路径引用
