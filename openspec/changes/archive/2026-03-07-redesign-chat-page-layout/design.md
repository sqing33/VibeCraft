## Context

当前 `#/chat` 页面已经具备完整的 chat 能力：会话创建/切换、流式回复、思考过程展示、附件发送与预览、手动压缩、分叉会话、归档删除等都已打通。实现集中在 `ui/src/app/pages/ChatSessionsPage.tsx`，页面本身已是左右分栏，但视觉重心偏工具型，消息列与输入区的节奏较紧，会话栏与主画布之间的层次也不够明确。

本次改动目标不是改变聊天协议或 store 行为，而是在前端布局层面将 Chat 页面重组为更接近现代 AI 工作台的沉浸式阅读界面。受影响范围主要在 `ChatSessionsPage.tsx`，必要时小幅调整 `App.tsx` 的 chat route 容器宽度约束；不引入新依赖，不修改 daemon API。

约束：
- 必须保留现有 chat 功能与交互能力，尤其是会话操作、附件、流式消息、思考过程、fork/compact/delete。
- 继续使用 HeroUI + Tailwind，避免引入额外布局框架。
- 应优先在单文件内完成布局重排，避免一次性拆分过多组件导致回归范围扩大。

## Goals / Non-Goals

**Goals:**
- 将 Chat 页面改为“左侧窄会话栏 + 中间大留白消息画布 + 底部稳定输入区”的信息架构。
- 收敛顶部信息密度，使消息阅读区成为主视觉焦点。
- 控制消息阅读列宽，避免长文本全宽铺开。
- 让输入区在主区底部稳定停靠，消息滚动与输入操作彼此独立。
- 保持现有聊天行为与数据流不变，主要修改布局和视觉层次。

**Non-Goals:**
- 不调整后端聊天 API、WebSocket 事件、store schema、会话数据结构。
- 不在本次引入右侧辅助面板、消息 hover 工具栏、虚拟列表等额外功能。
- 不重做全站 Topbar 或 Settings，只聚焦 `#/chat` 页面本身。

## Decisions

1) **保留现有数据流，只重排渲染结构**
- 方案：继续使用当前 `useChatStore`、`useDaemonStore` 与现有事件处理逻辑，仅调整 JSX 层级与 className。
- 备选：拆分新页面组件并重写消息/会话子组件。
- 取舍：先控制改动面，优先验证布局方向；后续若效果稳定，再做组件抽离。

2) **会话栏保持固定窄宽度，主画布使用居中可读列**
- 方案：外层维持两栏布局，左栏固定宽度（约 240~280px），右栏撑满；消息区内部再用 `max-w-*` 限制实际阅读列宽。
- 备选：消息区直接铺满右侧区域。
- 取舍：图片一的核心质感来自“宽容器 + 窄阅读列”的空间关系，直接铺满会损失沉浸感。

3) **输入区固定在主区底部，而不是跟随消息高度自然流动**
- 方案：主区采用 `flex-col`，消息滚动区 `flex-1 overflow-y-auto`，composer 区域 `shrink-0` 固定在底部。
- 备选：让输入区继续作为消息区尾部内容自然排列。
- 取舍：固定输入区能提升页面稳定性，更接近目标参考图的观感。

4) **保留消息气泡语义，但降低卡片感与边框噪音**
- 方案：消息仍区分 user/assistant，但减少过强边框和整页卡片包裹感，更多依靠留白和对齐建立层次。
- 备选：完全扁平化、去掉所有 bubble。
- 取舍：现有消息元数据（附件、thinking、token usage）较多，完全去 bubble 反而不利于组织复杂内容。

## Risks / Trade-offs

- [单文件重排导致 JSX 变动较大] → 通过保持现有状态逻辑不动、优先修改外层容器与样式类来降低风险。
- [侧栏变窄后会话操作显得拥挤] → 优先保留高频操作；若空间不足，可通过 icon-only 或二级操作收敛。
- [消息列过窄影响附件/代码展示] → 保留长内容自动扩展策略，对代码块/表格继续使用更宽容器。
- [输入区固定后小屏空间变少] → 通过 `overflow-hidden + 内部滚动` 保证主区可用，必要时在窄屏回退为单列堆叠。
