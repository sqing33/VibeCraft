## Context

当前应用对所有路由统一渲染 `Topbar`，聊天页自身再渲染「左侧会话卡片 + 右侧对话卡片」。这让聊天场景同时存在全局顶部导航和局部侧栏，视觉上更像“页面里的页面”，不符合这次想要的 Linear 风格工作台：左栏是页面本身的一部分，右侧才是主要内容面板。

## Goals / Non-Goals

**Goals:**
- 让 `#/chat` 进入独立的左右工作台布局，不再显示全局顶部栏。
- 让左侧栏承担页面导航、会话列表、系统状态和快捷操作。
- 保持右侧消息区的大部分现有实现不变，只重排头部结构。
- 保持消息流的可读宽度和输入区底部锚定能力。

**Non-Goals:**
- 不改动聊天消息渲染样式、发送逻辑、附件逻辑。
- 不重做其他路由（如 Orchestrations / Repo Library）的顶层布局。
- 不引入新的 UI 依赖或新的全局状态模型。

## Decisions

### 1. 仅对聊天路由隐藏全局 `Topbar`
- 决定：在 `App.tsx` 中对 `chat` 路由关闭顶部栏渲染，并让聊天页占满原先由顶部栏预留的高度。
- 原因：这是最小风险的路由级改动，不会影响其他页面既有导航结构。
- 备选方案：抽象出统一 App Shell 并让所有页面共用。未采用，因为这次只需要重排聊天页，整体壳层重构成本过高。

### 2. 左侧栏采用“页面融合式”而非“独立卡片式”视觉
- 决定：移除聊天页左栏外层圆角边框与卡片背景，只保留页面内边距、分区和滚动区域。
- 原因：更贴近目标参考图，也能强化右侧对话卡片的主内容地位。
- 备选方案：保留现有圆角卡片，仅把顶部栏内容挪入。未采用，因为仍会保留“页面里再套一个左栏卡片”的割裂感。

### 3. 右侧头部改为三段式布局
- 决定：头部使用三栏布局，左侧放 session id / 模型信息，中间放居中的标题，右侧放状态和分叉按钮。
- 原因：满足用户对“标题居中、信息在左、操作在右”的明确要求，同时减少上下堆叠造成的头部高度浪费。
- 备选方案：继续使用左侧上下堆叠标题+元信息。未采用，因为不符合这次新布局目标。

### 4. 复用现有会话列表和消息区逻辑，仅重排容器
- 决定：不拆动消息渲染、选择会话、发送消息等逻辑，只调整最外层 grid、左栏分区和右侧 header。
- 原因：把修改范围收敛在布局层，更易验证，也更不容易引入行为回归。

## Risks / Trade-offs

- [窄屏空间更紧] → 保留现有响应式单列/双列切换，在小屏下允许纵向堆叠。
- [顶部栏逻辑在聊天页出现少量重复] → 只复用少量状态文案和按钮组合，优先降低本次改动风险；后续若更多页面采用左 rail，再统一抽象。
- [居中标题受左右内容长度影响] → 使用三栏布局和文本截断，避免标题被挤压后换行破坏头部高度。
