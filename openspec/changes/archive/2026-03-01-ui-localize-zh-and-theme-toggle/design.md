## Context

当前前端已经完成工作流主路径（Kanban + 详情页），但文案仍存在中英文混杂，且没有主题切换能力。现有样式体系基于 Tailwind + shadcn，已具备通过 CSS 变量和 `dark` class 做主题切换的基础条件。本次变更仅涉及 UI 层，不调整后端 API 与数据结构。

## Goals / Non-Goals

**Goals:**

- 将 UI 可见文案统一为中文，覆盖主流程页面与常见状态/错误提示。
- 增加明暗主题切换开关，支持“浅色 / 深色”两种主题。
- 默认主题为浅色，并将用户选择持久化到本地存储。
- 保持当前功能链路不变，确保现有 API 交互行为不回归。

**Non-Goals:**

- 不引入 i18n 多语言框架（本次仅做中文单语落地）。
- 不改动后端接口、数据库模型与调度语义。
- 不做大规模视觉重设计（仅做主题色与文案层面的必要调整）。

## Decisions

1. 主题状态集中管理在 UI 侧轻量 store/hook 中，避免散落在页面组件中。
   - 备选方案：仅用组件局部 state；缺点是跨页同步与持久化复杂。
2. 主题切换采用 `document.documentElement.classList.toggle('dark')` + localStorage 持久化。
   - 备选方案：CSS 媒体查询自动跟随系统；不满足“默认浅色”的产品要求。
3. 中文化优先覆盖“用户可见路径”文案（标题、按钮、空态、错误、状态标签、设置项）。
   - 备选方案：一次性覆盖全部开发调试字符串；收益较低且风险较高，放到后续迭代。
4. DevTools 保持开发环境可见、生产隐藏规则不变，仅将其可见文案同步中文化。

## Risks / Trade-offs

- [文案遗漏风险] 某些边缘错误信息仍可能是英文 → 通过全局检索与页面走查补齐。
- [主题对比度风险] 浅色/深色下部分组件可读性不一致 → 对关键页面做明暗两套人工检查。
- [状态持久化风险] 本地存储异常导致主题恢复失败 → 提供浅色默认值并在异常时回退默认。

## Migration Plan

1. 新增主题 store/hook，接入 Topbar 主题开关并实现默认浅色。
2. 批量替换 UI 主流程文案为中文。
3. 在开发与生产构建分别验证主题开关和文案展示。
4. 完成后归档 OpenSpec 变更并同步基线 specs。
