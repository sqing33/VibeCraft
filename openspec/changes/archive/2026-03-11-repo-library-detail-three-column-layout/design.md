## Context

当前 Repo Library 仓库详情页仍以纵向信息堆叠为主：报告正文常驻首屏，快照和分析运行采用卡片列表，知识卡片和证据位于下半区。这个结构对“第一次看报告”尚可，但对已经完成解析、需要在多个卡片之间高频切换的知识库使用场景不友好。

同时，正式报告第一部分“技术栈与模块语言”的结构本身较稳定，已经包含仓库级上下文摘要，但目前只作为 markdown 文本存在，无法作为 UI 的结构化信息复用。

本次改动横跨后端报告抽取、仓库详情 API 和前端布局交互，适合先固定设计决策再实施。

## Goals / Non-Goals

**Goals:**
- 将仓库详情页改为适合持续浏览卡片的固定三栏工作台。
- 将快照和分析运行改为下拉选择器，压缩首屏控制区。
- 从正式报告中结构化提取技术栈摘要，并在仓库详情页左栏展示。
- 将完整报告改为显式查看动作，不再常驻首屏。
- 为中栏卡片列表和右栏 evidence 提供独立滚动区域，避免整页滚动干扰。

**Non-Goals:**
- 不重写报告生成 prompt 或正式报告的整体章节格式。
- 不改变卡片抽取和搜索结果的主语义模型。
- 不在本次变更中引入新的搜索过滤器、卡片分组或批量操作。
- 不新增独立的技术栈数据库表；首版使用仓库详情/快照详情中的派生字段即可。

## Decisions

### 1. 技术栈摘要由后端在正式报告解析阶段提取，而不是前端临时正则解析

原因：
- 报告摘要属于仓库/快照级派生数据，不是纯展示细节。
- 前端每次从 markdown 重复解析，规则分散且难以测试。
- 后端抽取后可被详情页、后续搜索筛选或其它页面复用。

方案：
- 在 `formal_report_extractor` 附近新增一个轻量的仓库摘要解析器，面向正式报告第一部分提取：
  - `generated_at`
  - `stack_overview`
  - `backend_summary`
  - `frontend_summary`
  - `other_modules_summary`
- 将这些字段挂到 repository detail 响应中的 snapshot/repository 上下文对象，优先以所选 snapshot 对应值返回。

备选方案：
- 前端直接解析 markdown。实现快，但规则重复、不可复用，放弃。

### 2. 报告正文不再常驻主页面，改为按钮触发的阅读弹层

原因：
- 解析完成后，用户更关注卡片浏览和证据下钻，而不是整篇报告全文。
- 报告常驻会显著挤压卡片与详情空间。

方案：
- 详情页顶部操作区增加 `查看报告`。
- 点击后打开右侧抽屉或大尺寸弹窗展示完整 markdown 报告。
- 首版优先使用当前 UI 组件库中易落地的 modal/dialog 方案；只要能稳定滚动阅读即可。

备选方案：
- 保留报告正文但默认折叠。仍占首屏结构和状态管理，不如直接外移。

### 3. 仓库详情页采用固定三栏布局，页面本体不滚动

原因：
- 仓库上下文、卡片列表、卡片详情是三种不同浏览任务，应该稳定并列。
- 整页滚动会导致卡片切换时视线和上下文频繁丢失。

方案：
- 内容区改为固定高度三栏 flex/grid：
  - 左栏：快照下拉、分析运行下拉、技术栈摘要、分析 Chat/同步动作。
  - 中栏：卡片列表，整栏内容区滚动。
  - 右栏：顶部固定卡片详情，底部 evidence 区滚动。
- 页面外层使用可视区剩余高度，内部通过 `min-h-0` + `overflow-hidden/auto` 控制滚动责任。

备选方案：
- 继续两栏加内嵌折叠块。会继续牺牲卡片浏览效率，放弃。

### 4. 快照与分析运行改为选择器，但保留现有数据模型

原因：
- 当前左栏不需要展示多条分析记录的大卡片列表，只需要上下文选择。
- 数据量不大，仍可直接从当前 detail payload 渲染，无需新增分页或二级 API。

方案：
- 使用下拉组件展示 snapshot/analysis options。
- 选中项变化时继续复用现有刷新/联动逻辑。
- 在选择器下方展示当前 analysis 的 chat/runtime 关键信息与动作按钮。

备选方案：
- 使用 tabs 或 segmented control。选项长度和数量更不稳定，不如下拉稳妥。

## Risks / Trade-offs

- [Risk] 正式报告第一部分标题或条目漂移会导致摘要提取失败 → Mitigation：解析器允许轻微标签漂移，并在提取失败时回退为空字段，不阻断卡片展示。
- [Risk] 三栏固定高度在小屏窗口中可能过于拥挤 → Mitigation：保留响应式断点，在较窄宽度下退化为上下布局或双栏堆叠。
- [Risk] 报告弹层可能让用户丢失当前卡片上下文 → Mitigation：报告只读展示，不刷新当前页状态；关闭后保留当前选择。
- [Risk] 新增结构化摘要字段但未持久化，未来可能重复解析 → Mitigation：首版允许在 detail 构建阶段从 snapshot report 派生，后续若复用面扩大再考虑持久化。

## Migration Plan

1. 后端增加正式报告技术栈摘要解析器，并把结构化字段挂到 detail API 返回模型。
2. 前端重构仓库详情页为三栏布局，保留现有卡片和 evidence 数据获取接口。
3. 用 `查看报告` 替换正文常驻区，验证 markdown 仍可完整查看。
4. 在已有分析数据上直接验证，不要求重跑仓库分析。

## Open Questions

- 首版报告查看控件使用 Modal 还是 Drawer：实现上以组件库可用性和稳定性为准。
- 技术栈摘要是否需要落库：本次先不强制，若后续需要检索/过滤再提升为持久化字段。
