## Context

Repo Library 的自动分析链路目前是：准备仓库 → 让 AI 输出最终 Markdown 报告 → 直接把该回复写入 `report.md` → 调用 Python 抽卡与搜索刷新。现有提示词虽然声明了模板，但系统只做了一个非常粗的“像正式报告”判断，无法保证报告能被下游消费。

这次问题说明仅靠提示词不可靠：AI 可能输出看似正确、但标题层级与 feature 组织方式不符合抽取链路预期的报告，导致 `cards.json` 与搜索索引都为空。

## Goals / Non-Goals

**Goals:**
- 让正式报告必须通过机器校验后才能进入 Repo Library 正式链路。
- 让校验失败可自动驱动 AI 重写，而不是让用户手动修补。
- 让自动分析与手动同步共享同一校验逻辑。
- 提供足够的失败工件，便于排障和人工复核。

**Non-Goals:**
- 不引入新的前端入口。
- 不改数据库 schema。
- 不把所有自然语言输出都限制为同一模板，仅限制“正式报告版本”。

## Decisions

### 1. 新增独立 `validate-report` 脚本，而不是把规则散落在 Go 里
- 选择：在 `services/repo-analyzer/app/` 新增校验入口，并通过 CLI 给 Go 调用。
- 原因：校验与抽卡都在 Python 侧，放在同一层可以直接复用 `build_cards_payload(...)`，避免 Go/Python 双份规则漂移。
- 备选方案：在 Go 中重写一份 Markdown 校验与抽卡预检；放弃，因为重复实现成本高且更易失配。

### 2. 校验通过标准使用“至少可抽出 1 张卡片”
- 选择：除了结构检查，还要求 `card_count > 0`。
- 原因：Repo Library 最关键的下游就是卡片与搜索；只看标题结构会放过“格式像但内容不可消费”的报告。
- 备选方案：只验证标题与段落；放弃，因为无法真正防止空结果入库。

### 3. 自动重写在同一 chat session 中进行，并把错误清单回灌给 AI
- 选择：校验失败后，继续在同一分析会话中追加一个修订提示，要求 AI 返回完整修订版最终报告。
- 原因：同一会话保留上下文、成本低，且方便用户后续继续追问。
- 备选方案：后端本地自动修正文档；放弃，因为结构错误往往伴随内容组织错误，机械修补风险高。

### 4. 重试上限为 5 次，失败后保留草稿但不入正式知识库
- 选择：每次尝试都保存 `report.attempt-N.md` 和对应校验 JSON；若 5 次后仍失败，则保留最后一版 `report.invalid.md` 与 `report.validation.json`，并停止抽卡/索引刷新。
- 原因：避免无限重试，同时保留足够诊断信息。
- 备选方案：继续把最后一版当正式报告写入；放弃，因为这会继续污染 Repo Library。

### 5. 按 feature 逐项输出，而不是把所有关注点拼成一个“功能 1”
- 选择：Prompt 强制 `run.Features` 一项对应一项功能块，并在 AI 细节部分保持同序映射。
- 原因：这能让验证器稳定核对 feature 数量，也让“需要表格”的要求落到具体 feature。
- 备选方案：允许合并多个 feature 到一个功能块；放弃，因为难以验证且更容易丢字段。

## Risks / Trade-offs

- [AI 多次仍无法满足格式] → 保留无效草稿与校验结果，避免正式数据被污染。
- [严格校验导致部分原本可读的报告被拒绝] → 以“可抽卡”为最终门槛，确保拒绝是有下游价值的。
- [多轮重试增加分析耗时] → 只在校验失败时触发重试，并限制 5 次上限。
- [Prompt 与 validator 再次漂移] → 让 validator 成为唯一裁决者，并为真实失败样本补回归测试。
