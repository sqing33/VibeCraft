# AI Agent Orchestrator（vibe-tree）详细规划（MVP → v1）

> 目标：把“多模型/多工具 AI 编排”做成一套本地化、可视化、可干预的工作台。形态类似“可视化 AI tmux”：左侧拓扑编排，右侧并发终端日志，顶层用 Kanban 管工作流状态。

本文件是**可直接开工的技术规划/产品规格**：包含范围、术语、架构、协议、数据模型、状态机、UI、里程碑、测试与风险。  
（已知偏好：**Ubuntu 优先**，Windows 后续适配；前端选 **React**）

---

## 0. 成功标准（Success Criteria）

MVP 达成即算成功（必须全部满足）：

1. 能创建一个 Workflow（卡片），指定工作目录（workspace），点击 Start 后启动 “Master 规划节点”。
2. Master 产出一份结构化 DAG（节点 + 依赖 + 每个节点的执行器/专家 + Prompt）。
3. 在 **手动审批模式** 下，DAG 子节点停在 `pending_approval`，用户可改 Prompt/改专家后统一批准并发执行。
4. 后端用 PTY 拉起多个并发命令，前端用 xterm.js **实时渲染 ANSI 彩色日志**；并可点选左侧节点联动右侧对应终端。
5. 任意节点可 Cancel；Cancel 后能可靠终止进程并更新状态，且日志不丢（至少落盘可回放）。
6. 进程/daemon 重启后：Workflow/Node 元数据可从 SQLite 恢复；日志可从文件回放；UI 能重新订阅并看到正确状态。
7. 任务记录可追溯：Workflow/Node 的关键输入（prompt/专家选择/审批修改）、状态变更、执行结果摘要可在 UI 中回放，不依赖 CLI 自身历史；支持基础检索（至少按标题/节点标题/摘要）。

## 1. 范围定义（Scope）

### 1.1 MVP 做什么
- 本地单用户、单机编排（默认仅 `127.0.0.1`）。
- Web UI（React SPA）+ 桌面壳（Wails）两种入口。
- 工作流（Workflow）+ DAG 节点（Node）+ 执行尝试（Execution）。
- 执行器（Expert/Runner）以“SDK 驱动”为主：`bash`（兜底/可测基线）、`codex`（OpenAI SDK）、`claudecode`（Anthropic SDK）。
- 两种运行模式：
  - **Auto**：规划后自动启动全部节点（遵循依赖）。
  - **Manual approval**：规划后挂起，用户审核/修改后启动。
  - **运行中动态切换**：支持在详情页把 Auto 切到 Manual（执行阻断/断点），用于拦截后续节点。
- 日志与终端：
  - 后端：PTY + 文件落盘 + WebSocket 推送
  - 前端：xterm.js 渲染；多终端并排/横向滚动
- 任务记录：保存 workflow/node 的 prompt、审批修改、执行结果摘要与结构化结果；原始日志落盘可回放（v1 增加全文检索）。
- DAG 可视化：React Flow + 自动布局（dagre）。
- 存储：SQLite（元数据）+ 文件（高频日志与执行产物）。

### 1.2 MVP 不做什么（明确排除）
- 多用户、权限系统、云同步、团队协作。
- 远程 Agent（SSH/容器/多机）。
- 插件市场/在线下载专家。
- 完整的“语义路由模型”与自动选择：MVP 先靠 Master 输出 + 用户可改。
- 复杂 DAG 条件分支（if/else）、循环、动态并发扩缩；MVP 只做依赖图（DAG）。

## 2. 核心术语与对象模型（Glossary）

**Workflow（工作流/卡片）**  
用户创建的一次“要完成的任务”。对应 Kanban 卡片：`todo / running / done / failed`。

**Node（节点）**  
Workflow 的一个执行步骤，可是 `master`（规划/拆解）或 `worker`（执行）。

**Edge（依赖边）**  
表达节点之间的依赖关系：`from -> to`，表示 `to` 需要等待 `from` 成功（MVP）。

**Execution（执行尝试）**  
节点的一次运行记录（支持 retry、racing、手动重跑）。一个 Node 可以有多个 Execution。

**Expert（专家/执行器配置）**  
一个可执行的专家配置（`run_mode=oneshot` 启动本地命令，或 `run_mode=sdk` 直接调用 OpenAI/Anthropic SDK），包含模型/命令、环境变量、超时策略、输出结构化约束等。

**Runner（运行时适配层）**  
后端内部接口：负责启动进程（PTY/pipes）、流式读日志、写入输入（可选）、cancel/kill、退出码采集。

**Event（事件/审计记录）**  
记录系统内发生的关键动作与状态变更（例如：状态变更、prompt 修改、DAG 生成、execution 退出）。用于“任务记录/回放/搜索”的主干数据。

**Session（专家会话/预热实例）**  
某个 expert（尤其是 `run_mode=daemon`）的常驻进程实例，可被多个 Node 的 Execution 复用以避免冷启动；也是“一个终端一个 CLI 对话”的主要承载体（并发时通过会话池扩展）。

## 3. 架构（Localhost-first + 瘦客户端）

### 3.1 组件拆分

1. **Go Daemon（核心引擎）**
   - 常驻后台
   - 任务编排与状态机
   - PTY/进程管理
   - SQLite 与日志文件读写
   - HTTP API + WebSocket

2. **Web UI（React SPA）**
   - Kanban 列表页
   - Workflow 详情页（DAG + Terminal Pool）
   - 仅通过 HTTP/WS 通信，不做本地 IPC 绑定

3. **Desktop Shell（Wails）**
   - 启动/托管 Go Daemon
   - 打开 WebView 加载 `http://127.0.0.1:<port>/`
   - 提供系统托盘、开机自启（v1 之后）、文件选择对话框等

4. **Experts（SDK/CLI）**
   - `bash`：用于 MVP 调试与可测性（必备 fallback）
   - `codex` / `claudecode`：作为“专家模型/执行端”（SDK 驱动，按配置启用）

### 3.2 网络与安全默认值（MVP）

- Daemon 默认只监听：`127.0.0.1:<port>`
- **LAN 访问**作为显式开关（v1 或 MVP+）：
  - 开启时监听 `0.0.0.0`
  - 必须启用 `Bearer token`（首次生成、可重置、可过期）
  - UI 与 API 统一走同一 auth 机制

### 3.3 数据流（简化）

1. UI 创建 Workflow → POST `/api/v1/workflows`
2. UI Start → POST `/api/v1/workflows/{id}:start`
3. Daemon 创建 master Node + Execution，启动 Expert（如 `codex` 或 `bash`）
4. master 输出 DAG JSON → Daemon 校验 + 写入 SQLite → WS 推送 `dag.generated`
5. Manual 模式下：UI 修改 Node prompt/专家 → PATCH `/api/v1/nodes/{id}`
6. UI Approve → POST `/api/v1/workflows/{id}:approve` → Daemon 按依赖并发启动 worker Nodes
7. 运行日志：Runner 写文件 + WS 推送 `node.log`；UI 用 xterm.js 渲染

- DAG：React Flow + dagre
- Terminal：xterm.js（必要时启用 WebGL addon）

### 4.3 桌面端（Wails）
- Wails v2（Go + WebView）
- 模式：Wails 启动 daemon + 打开 `127.0.0.1` URL

### 4.4 仓库结构（建议，MVP）

> 当前仓库只有规划文档；落地代码时建议按“后端/前端/桌面壳”拆成三块，避免互相污染构建链路。


## 5. 运行时/执行器设计（Expert + Runner）

### 5.1 Expert 配置（MVP 约定）

配置存储位置（Ubuntu）：  
- config：`$XDG_CONFIG_HOME/vibe-tree/config.json`（默认 `~/.config/vibe-tree/config.json`）  
- data：`$XDG_DATA_HOME/vibe-tree/`（默认 `~/.local/share/vibe-tree/`）

`config.json` 示例（MVP）：

```json
{
  "server": { "host": "127.0.0.1", "port": 7777 },
  "execution": { "max_concurrency": 6, "kill_grace_ms": 1500 },
  "experts": [
    {
      "id": "bash",
      "label": "Bash",
      "run_mode": "oneshot",
      "command": "bash",
      "args": ["-lc", "{{prompt}}"],
      "env": {},
      "timeout_ms": 1800000
    },
    {
      "id": "codex",
      "label": "Codex (OpenAI SDK)",
      "run_mode": "sdk",
      "sdk": { "provider": "openai", "model": "gpt-5-codex", "max_output_tokens": 8192 },
      "env": { "OPENAI_API_KEY": "${OPENAI_API_KEY}" },
      "timeout_ms": 3600000
    },
    {
      "id": "claudecode",
      "label": "ClaudeCode (Anthropic SDK)",
      "run_mode": "sdk",
      "sdk": { "provider": "anthropic", "model": "claude-3-7-sonnet-latest", "max_output_tokens": 4096 },
      "env": { "ANTHROPIC_API_KEY": "${ANTHROPIC_API_KEY}" },
      "timeout_ms": 3600000
    }
  ]
}
```

> 说明：`{{prompt}}` 为模板占位；`${ENV}` 表示从系统环境变量注入。`run_mode=sdk` 不再依赖外部 CLI，直接调用 provider 官方 SDK；模型与输出策略由 `sdk.*` 字段控制。

#### `run_mode`（性能优化：oneshot vs daemon/mcp）

- `oneshot`（MVP 必做）：每个 Node/Execution 启动一个新进程，退出即结束。适合 `bash` 与其他本地命令工具。
- `sdk`（MVP 支持）：不启动外部进程，直接调用 OpenAI/Anthropic 官方 SDK，输出仍按 Execution 维度落盘/推送（支持流式输出与取消）。
- `daemon`（v1 目标，MVP 可先预留字段）：专家进程常驻复用，用于 **MCP/LSP/语言服务器**这类冷启动昂贵的工具链。
  - 运行时要点（v1）：
    - Daemon 进程按 `expert_id` 维度维护单例或池（默认单例）。
    - Node 执行变为“对 daemon 的一次 call”，输出仍按 Execution 维度落盘/推送。
    - 支持 `idle_ttl_ms`（空闲自动退出）与崩溃自愈（自动重启 + 重试策略）。
  - 配置字段（v1）建议补充：
    - `daemon`: `{ "idle_ttl_ms": 300000, "startup_timeout_ms": 15000 }`
    - `protocol`: `"mcp-stdio"` 或 `"jsonrpc-stdio"`（先从 stdio 协议做起）

### 5.2 Runner 接口（后端内部）

MVP 必须支持：
- PTY 模式（默认）：保证 ANSI 与进度条
- Cancel：先 SIGTERM，超时后 SIGKILL
- 流式读：stdout/stderr 或统一 PTY 输出
- 落盘：边读边 append 到 log 文件

Runner 伪接口：

```go
type Runner interface {
  StartOneshot(ctx context.Context, spec RunSpec) (ProcessHandle, error) // run_mode=oneshot（MVP）
  // DaemonCall(ctx, spec) (CallHandle, error)                           // run_mode=daemon（v1）
}

type ProcessHandle interface {
  PID() int
  Wait() (ExitResult, error)
  Cancel(grace time.Duration) error
  WriteInput(p []byte) (int, error) // MVP 可不启用
}
```

### 5.3 上下文修剪（Context Pruning）策略（MVP）

MVP 原则：**少而稳定，不做复杂“记忆”**。

- 每个 Node 执行时，Daemon 只注入：
  1) Workflow 标题/目标
  2) Node prompt（用户可编辑）
  3) workspace 路径
  4) 可选：从用户选择的文件列表读取内容（v1）
- 不维护长对话历史；Master/worker 只需要“当前任务 + 必要输入”。

### 5.4 会话复用与预热（Session Pool，解决 CLI/MCP 冷启动）

你提到的“切换 CLI/初始化环境/MCP 很慢”和 VSCode 插件那种“新问题立刻开始、旧问题后台继续”的体验，本质需要两点：

1) **复用**：把昂贵的初始化从“每次 Execution 启动”挪到“少量常驻 Session”。  
2) **并发**：同一个 Session 通常只能顺序处理请求；要做到“旧任务继续跑、新任务立刻跑”，需要**会话池**（多个 Session 并行）。

落地策略（MVP → v1）：

- `run_mode=oneshot`（MVP）：每个 Execution 启动新进程。体验简单、隔离强，但冷启动成本高。
- `run_mode=daemon`（v1）：为某些 expert（尤其 MCP/LSP/语言服务器）启动常驻进程作为 Session：
  - 由后端 `SessionManager` 维护 `expert_id` 维度的 Session 池：`min/max/idle_ttl_ms`。
  - 调度时按需分配空闲 Session；若都忙且未达 `max`，则新建 Session；否则排队等待。
  - UI 展示方式建议：
    - MVP：仍按“一个 Execution 一个终端 pane”展示（实现最简单）。
    - v1：增加“按 expert/session 分组视图”，实现更像“一个终端一个 CLI 对话”的体验。

配置建议（v1）：
- expert 增加 `pool`：`{ "min": 1, "max": 3, "idle_ttl_ms": 300000 }`
- expert 增加 `warmup`（可选）：启动后执行一条初始化命令/握手（例如加载 workspace、连接 MCP server），把冷启动成本前置。

关键约束（写进实现说明，避免误解）：
- **单个交互式 CLI 进程无法同时执行两个请求**；VSCode 的“同时问新问题”往往是后台起了多个 session/agent。这里用“会话池”实现同等效果。

## 6. 状态机与调度（决定性细节）

### 6.1 Workflow 状态
- `todo`：新建未开始
- `planning`：master 运行中
- `pending_approval`：已生成 DAG，等待用户批准（manual 模式）
- `running`：worker 节点执行中
- `done`：全部终止且成功（或用户手动标记完成，v1）
- `failed`：有节点失败且策略为 fail-fast（MVP 默认）
- `canceled`：用户取消

### 6.2 Node 状态（MVP）
- `created`
- `pending_approval`（manual 模式）
- `queued`（等待依赖/等待并发额度）
- `running`
- `success`
- `error`
- `canceled`

允许的状态转移（MVP）：
- `created -> pending_approval | queued`
- `pending_approval -> queued | canceled`
- `queued -> running | canceled`
- `running -> success | error | canceled`
- `error -> queued`（retry）

### 6.3 调度策略（MVP）
- 默认 `fail-fast`：任一节点 `error` → Workflow 标记 `failed`，其余 `queued` 节点不再启动（已 running 的允许继续或取消：MVP 默认继续跑完并标记）。
- 并发上限：`execution.max_concurrency`（默认 6）
- 依赖满足：节点所有入边 `from` 必须 `success` 才可入队

### 6.4 执行阻断开关（Execution Breakpoint Toggle，MVP 建议纳入）

> 目的：Auto 运行时发现不对劲，可立刻切到 Manual，阻断后续节点启动（避免“越跑越错”）。

- Workflow 的 `mode`（`auto|manual`）**允许运行中动态修改**。
- 从 `auto -> manual`：
  - 调度器停止启动新节点；
  - 所有尚未 `running` 的 `queued` 节点转为 `pending_approval`（已满足依赖但未启动的也要被拦截）；
  - 未来变为 runnable 的节点，直接进入 `pending_approval`。
- 从 `manual -> auto`：
  - 调度器将所有“依赖已满足”的 `pending_approval` 节点转为 `queued` 并继续调度（受并发上限约束）。

### 6.5 重启恢复（MVP）
- Daemon 启动时扫描 DB：
  - 所有 `running` 的 Execution 统一标记为 `error`（原因：daemon 重启导致无法保证进程仍在；MVP 采取保守策略）
  - 节点状态回滚到 `error`，允许用户手动 retry
- 日志回放：前端可按需拉取 log 文件尾部（HTTP）或 WS 订阅实时流

## 7. 存储设计（SQLite + 文件）

### 7.1 文件目录布局（Ubuntu）

`$XDG_DATA_HOME/vibe-tree/`（默认 `~/.local/share/vibe-tree/`）：
- `state.db`：SQLite
- `logs/<workflow_id>/<node_id>/<execution_id>.log`：ANSI 日志
- `artifacts/<workflow_id>/...`：执行产物（可选）

### 7.2 SQLite 表（MVP 建议）

> 注意：为支持真正 DAG，需要 `edges` 表；仅 `ParentNodeID` 不足。

#### SQLite 并发锁死预防（必须）

即使启用 WAL，在 Go 高并发写入下仍可能出现 `database is locked`。MVP 强制以下约束：

- DSN 强制 `_busy_timeout=5000`（单位毫秒）。
- Go 侧强制单连接池（单线程队列写入）：
  - `db.SetMaxOpenConns(1)`
  - `db.SetMaxIdleConns(1)`

GORM 初始化示例（概念）：

```go
dsn := "file:" + stateDBPath + "?_pragma=journal_mode(WAL)&_pragma=synchronous(NORMAL)&_busy_timeout=5000&_foreign_keys=on"
gdb, _ := gorm.Open(sqlite.Open(dsn), &gorm.Config{})
sqlDB, _ := gdb.DB()
sqlDB.SetMaxOpenConns(1)
sqlDB.SetMaxIdleConns(1)
```

补充约束：
- 禁止在事务内做任何外部 IO（网络/PTY），确保写锁持有时间极短。

```sql
-- workflows
CREATE TABLE workflows (
  id TEXT PRIMARY KEY,
  title TEXT NOT NULL,
  workspace_path TEXT NOT NULL,
  mode TEXT NOT NULL,            -- auto | manual（可运行中切换）
  status TEXT NOT NULL,          -- todo | planning | pending_approval | running | done | failed | canceled
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL,
  error_message TEXT,
  summary TEXT
);

-- nodes
CREATE TABLE nodes (
  id TEXT PRIMARY KEY,
  workflow_id TEXT NOT NULL,
  node_type TEXT NOT NULL,       -- master | worker
  expert_id TEXT NOT NULL,       -- bash | codex | ...
  title TEXT NOT NULL,
  prompt TEXT NOT NULL,
  status TEXT NOT NULL,          -- created | pending_approval | queued | running | success | error | canceled
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL,
  last_execution_id TEXT,
  result_summary TEXT,
  result_json TEXT,
  error_message TEXT,
  FOREIGN KEY(workflow_id) REFERENCES workflows(id)
);

-- edges
CREATE TABLE edges (
  id TEXT PRIMARY KEY,
  workflow_id TEXT NOT NULL,
  from_node_id TEXT NOT NULL,
  to_node_id TEXT NOT NULL,
  source_handle TEXT,            -- React Flow: sourceHandle（可空）
  target_handle TEXT,            -- React Flow: targetHandle（可空）
  type TEXT NOT NULL,            -- success (MVP)
  FOREIGN KEY(workflow_id) REFERENCES workflows(id)
);

-- executions
CREATE TABLE executions (
  id TEXT PRIMARY KEY,
  node_id TEXT NOT NULL,
  attempt INTEGER NOT NULL,
  pid INTEGER,
  exit_code INTEGER,
  status TEXT NOT NULL,          -- running | success | error | canceled
  log_path TEXT NOT NULL,
  started_at INTEGER NOT NULL,
  ended_at INTEGER,
  error_message TEXT,
  FOREIGN KEY(node_id) REFERENCES nodes(id)
);

-- events（审计事件；不要把高频 log chunk 存进来）
CREATE TABLE events (
  id TEXT PRIMARY KEY,
  workflow_id TEXT NOT NULL,
  entity_type TEXT NOT NULL,      -- workflow | node | execution
  entity_id TEXT NOT NULL,
  type TEXT NOT NULL,             -- status.changed | prompt.updated | dag.generated | ...
  ts INTEGER NOT NULL,
  payload_json TEXT NOT NULL,
  FOREIGN KEY(workflow_id) REFERENCES workflows(id)
);
CREATE INDEX idx_events_workflow_ts ON events(workflow_id, ts);
```

### 7.3 日志策略（MVP）
- 每个 Execution 一个 log 文件
- 写入：bufio + 定期 flush（例如 100ms 或按行）
- 轮转：MVP 先按 Execution 切文件即可；v1 再加 size/天轮转与清理策略
- WebSocket 推送：只推“增量 chunk”，并在 UI 侧做 backpressure（如果 UI 断开，靠文件回放）

### 7.4 任务记录与检索（不依赖 CLI 历史）

你提到“只靠各个 CLI 的记录不够”是对的：要保留可追溯的任务历史，系统必须在 CLI 之外建立一层**统一记录层**。

MVP 采用“三层记录”以同时满足**大数据量**与**高并发**：

1) **元数据（SQLite）**：Workflow/Node/Execution 的结构化字段（title、prompt、expert、状态、开始结束时间、退出码、result_json/result_summary…）。  
2) **审计事件（SQLite，append-only）**：记录关键动作与变更（状态变更、prompt 修改、DAG 生成、审批/阻断开关切换、执行开始/结束…），用于时间线回放与调试。  
3) **原始流（文件）**：高频日志/大量文本一律落盘（每 Execution 一文件），避免把 DB 写爆或锁住。

检索策略：
- MVP：按 workflow 标题/节点标题/结果摘要过滤（来自 SQLite 字段）。
- v1：引入 SQLite FTS5（对 title/prompt/result_summary 建索引），实现全局搜索；原始日志默认不入索引（必要时做“按需索引”或只索引 tail）。

## 9. Master 规划输出（DAG JSON Schema）

MVP 只要求 master 输出一个可解析的 JSON（由 daemon 校验）。

#### 语义路由与兜底规则（必须写入 Master Prompt）

- 默认选择“全能/通用”专家（由系统配置或用户选择的默认 expert 决定）。
- 仅当任务复杂度高、需要特定工具链/技能时，才切到专门专家。
- Master 必须输出“为什么选这个专家”（routing_reason），否则视为不合格规划（MVP 可先作为 warning，不 hard fail）。
- 若 expert 不存在/不可用：daemon 用 `fallback_expert_id` 回退；再不行用系统默认 expert 回退。

Schema（概念）：

```json
{
  "workflow_title": "string",
  "nodes": [
    {
      "id": "string",
      "title": "string",
      "type": "worker",
      "expert_id": "bash|codex|claudecode",
      "fallback_expert_id": "bash|codex|claudecode",
      "complexity": "low|medium|high",
      "quality_tier": "fast|balanced|deep",
      "model": "string|null",
      "routing_reason": "string",
      "prompt": "string"
    }
  ],
  "edges": [
    {
      "from": "node_id",
      "to": "node_id",
      "type": "success",
      "source_handle": "string|null",
      "target_handle": "string|null"
    }
  ]
}
```

约束（MVP）：
- 必须无环（daemon 校验）
- `expert_id` 必须在 config 的 experts 中存在；否则 reject 并提示用户修正
- 节点 `id` 仅需在该 DAG 内唯一（daemon 可重写为内部 ID）
- `complexity/quality_tier/model/routing_reason/fallback_expert_id` 可为空：为空时由 daemon 自动补默认值（例如 `complexity=medium`，`quality_tier=balanced`，`fallback_expert_id=默认expert`）

复杂度与“快/慢模型”的默认映射（MVP）：
- `complexity=low` → `quality_tier=fast`
- `complexity=medium` → `quality_tier=balanced`
- `complexity=high` → `quality_tier=deep`

实现建议：
- 对支持自动选型的工具（例如某些 CLI 会自动在 fast/deep 模型间切换）：可忽略 `quality_tier/model` 或把 `model` 设为 `"auto"`。
- 对不支持自动选型的工具：由 orchestrator 根据 `quality_tier` 映射到具体 `expert_id` 变体或 `model` 参数（见 5.1 的 expert 配置模板化能力）。

## 13. 风险与对策（提前写死）

1. **外部 AI SDK 调用不稳定**
   - 对策：MVP 强制支持 `bash` runner；AI runner 先做最小可用（一次性 prompt → 输出），再逐步支持更复杂交互。

2. **PTY 细节与跨平台差异（Windows ConPTY）**
   - 对策：Runner 抽象隔离；先 Ubuntu 产出稳定 API；Windows 作为独立里程碑实现 `conpty_runner`。

3. **日志量太大导致内存/前端卡顿**
   - 对策：后端只推增量，不缓存全量；前端 xterm 限制 scrollback；回放走 tail API。

4. **LAN 模式带来安全风险**
   - 对策：默认只 localhost；LAN 必须 token；明确“这是本地工具非云服务”。

5. **SQLite 高并发写导致锁冲突**
   - 对策：强制 WAL + `_busy_timeout=5000` + `db.SetMaxOpenConns(1)`，并避免长事务（详见 7.2）。

## 14. v1+ 方向（非 MVP）

- Model Racing（规划竞速）：多个 master 并发产出 DAG proposal，UI 对比选用
- 条件边/分支执行（if/else）
- 文件上下文选择器（从 workspace 选文件并注入到 prompt）
- 更强恢复：daemon 重启后尝试探测 PID 是否仍在、支持重新 attach
- `run_mode=daemon`：MCP/LSP 等常驻专家进程复用（毫秒级响应）
- 插件系统：专家 runner 支持脚本/二进制插件 + 版本管理

### 14.1 规划竞速（Plan Racing）实现草案

目标：创建 workflow 时可选择多个“规划候选”（不同 provider/不同模型档位），并发产出多份 DAG；用户选中最佳方案后再进入正式执行。

核心流程：
1. 创建 workflow（`status=todo`）。
2. 启动 plan_race：为每个 candidate 启动一个 master execution（并发上限同样受 `max_concurrency` 约束）。
3. 每个 candidate 输出 DAG JSON → daemon 校验后写入 `plan_proposals.dag_json`，并 WS 推送 `plan.proposal.ready`。
4. 用户在 UI 选择某个 proposal：
   - 把 proposal 的 DAG “提交”为正式 nodes/edges（进入现有执行流程：manual 或 auto）。
   - 取消其他 proposals（kill 对应 execution）。

数据模型（概念）：

```sql
CREATE TABLE plan_proposals (
  id TEXT PRIMARY KEY,
  workflow_id TEXT NOT NULL,
  expert_id TEXT NOT NULL,
  quality_tier TEXT,             -- fast|balanced|deep
  model TEXT,                    -- 可空/auto
  execution_id TEXT NOT NULL,    -- 关联 executions(id)
  status TEXT NOT NULL,          -- running|success|error|canceled|selected
  dag_json TEXT,
  error_message TEXT,
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL,
  selected_at INTEGER,
  FOREIGN KEY(workflow_id) REFERENCES workflows(id)
);
```

API/WS（概念）：
- `POST /api/v1/workflows/{id}:plan_race`（body: candidates 列表）
- `POST /api/v1/workflows/{id}:select_plan`（body: proposal_id）
- WS：`plan.proposal.started | plan.proposal.ready | plan.proposal.failed | plan.selected`
